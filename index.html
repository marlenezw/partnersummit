Meet Zava

A DIY retailer with a sales-data problem

✦ · ✦ · ✦

The problem

👤 Customers

Want to find products by feature, use case, or vague description — not exact SKU.

📊 Store managers

Want to analyze sales in plain English — only for their store.

🔒 Security

One database, many roles. We can't let the LLM leak someone else's data.

One agent. Different identities. Same database.

What we're building today

A conversational agent that:

  • Speaks natural language to a Postgres sales database
  • Finds products by meaning, not just keywords (pgvector)
  • Respects who is asking, at the database layer (RLS)
  • Generates charts, runs code, and searches the web when it helps
  • Runs locally, then deploys to Azure with one command

Agents, briefly

✦ · ✦ · ✦

An agent is a loop

A model that can reason, decide which tool to call, observe the result, and keep going until the goal is met.

🧠 Model

Plans & decides next step

🛠️ Tools

How it acts on the world

🔁 Loop

Until it has the answer

LangChain makes that loop tiny

from langchain.agents import create_agent
from langchain_openai import ChatOpenAI

model = ChatOpenAI(
    model="gpt-5-mini",
    use_responses_api=True,   # 👈 OpenAI Responses API
)

agent = create_agent(
    model=model,
    tools=mcp_tools,          # 👈 tools come from an MCP server
    prompt=system_prompt,
)

async for event in agent.astream({"messages": history}):
    ...   # stream tokens + tool calls back to the UI

A few lines — the rest is tools, data, and instructions.

Model Context Protocol

USB-C for LLM tools

✦ · ✦ · ✦

Why MCP?

Before

  • Tools tightly coupled to one framework
  • Re-implement for every model / SDK
  • Hard to share across teams

With MCP

  • One server — many clients
  • Standard tools, resources, prompts
  • Swap models without rewriting tools

The three primitives

🛠️ Tools

Functions the model can call.
execute_sales_query(...)

📄 Resources

Read-only context the model can fetch.
schema://retail

💬 Prompts

Reusable templates the host can offer the user.

Today we focus on tools — the workhorse.

Our MCP server (FastMCP)

from fastmcp import FastMCP, Context

mcp = FastMCP("Zava Sales Analysis Tools", lifespan=lifespan)

@mcp.tool(description="Run a SELECT against the Zava sales DB")
async def execute_sales_query(ctx: Context, query: str) -> str:
    rows = await db_provider.execute_query(query)
    return json.dumps(rows, default=str)

@mcp.tool(description="Find products by meaning, not keywords")
async def semantic_search_products(ctx: Context, query: str, k: int = 5):
    embedding = await embedding_provider.get_embedding(query)
    return await db_provider.vector_search(embedding, k)

One database. Two superpowers.

Row Level Security + pgvector

✦ · ✦ · ✦

Row Level Security — safety at the data layer

The agent is creative. The database is the bouncer.

-- Each store manager gets their own RLS identity
ALTER TABLE retail.orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY store_isolation ON retail.orders
  USING (store_id = current_setting('app.store_id')::uuid);

-- Per request, the MCP server sets the identity:
SET LOCAL app.store_id = '';
SELECT * FROM retail.orders;   -- 👈 only THIS store's rows

Even if the LLM writes SELECT *, it only sees rows it's allowed to see.

pgvector — semantic search, in your database

-- Products table has a 1536-dim embedding column
CREATE EXTENSION vector;
ALTER TABLE retail.products ADD COLUMN embedding vector(1536);

-- "outdoor lights for a kitchen" ➜ embedding ➜ nearest neighbours
SELECT product_name, 1 - (embedding <=> $1) AS similarity
FROM retail.products
ORDER BY embedding <=> $1
LIMIT 5;

We embed with text-embedding-3-small on Azure OpenAI — same auth, same identity, no API keys.

The OpenAI Responses API

Built-in tools, for free

✦ · ✦ · ✦

Tools you don't have to write

🖼️ Image generation

"Show me a chart of sales by category, in pastel."

🐍 Code interpreter

Run Python on the model's results — pivot, plot, stats.

🌐 Web search

Look up current pricing, competitors, weather, news.

Combine these with your MCP tools — one agent, every superpower.

Turning them on

tools = mcp_tools + [
    {"type": "image_generation"},
    {"type": "code_interpreter", "container": {"type": "auto"}},
    {"type": "web_search"},
]

agent = create_agent(model=model, tools=tools, prompt=system_prompt)

Built-ins run on OpenAI's side. MCP tools run on yours. The model picks.

One question, end to end

User: "Top selling outdoor lights last quarter for store 7?"

  • Agent → get_current_utc_date (fix "last quarter")
  • Agent → semantic_search_products("outdoor lights") via MCP
  • MCP embeds query, runs pgvector search, returns 5 candidates
  • Agent → get_table_schemas → execute_sales_query
  • Postgres applies RLS — only store 7's rows come back
  • Agent → image_generation → bar chart
  • Agent streams answer + chart back to the UI

# 4. Open the UI, ask a question

ENVIRONMENT=local → the agent uses langchain-mcp-adapters to connect to your local MCP server. In production, the Responses API connects to it directly.

The system prompt is half the agent

  • Role & tone — "Zava sales analyst, friendly, professional"
  • Tool choice rules — semantic search for vague names, exact SQL otherwise
  • Schema discipline — always call get_table_schemas before SQL
  • Output shape — markdown tables, LIMIT 20, no internal IDs
  • Safety — never invent data, never say what you don't know

Takeaways

  • Agents are loops — LangChain shrinks them to a few lines
  • MCP decouples tools from models — write once, reuse
  • Push safety down — RLS is your seatbelt when the LLM gets creative
  • pgvector beats keyword search for vague, human queries
  • Responses API gives you image, code & web tools for free
  • Identity, not keys — Entra ID end-to-end
